# BL4CK Rebrand — Change Log

This folder documents every change made during the Breeze → BL4CK rebrand.
Each entry explains **what** was changed and **how it works**, so the rebrand is
auditable and reversible.

**Scope decision (agreed):**
- Rebrand everything **users can see**: the web dashboard, and the downloaded
  installer / executables / Windows services (Task Manager, Services.msc,
  Add/Remove Programs, file Properties all show **BL4CK**).
- **Do NOT** touch invisible database internals (the Postgres role `breeze_app`
  and the row-level-security functions `breeze_has_org_access`, etc.). Nobody
  ever sees those strings; renaming them carries all the tenant-isolation risk
  for zero branding value. They stay exactly as-is so nothing breaks.
- Full plan: `.claude/plans/bl4ck-rebrand.plan.md`.

## Phase 2 — Installer / executables / services show BL4CK

Goal: after a user downloads and installs, **everything they see is BL4CK** —
process names in Task Manager, entries in Services.msc, the install folder,
Add/Remove Programs, and file Properties.

### What was renamed (token-scoped, whole agent tree + build/release/web)
| Surface | Before → After |
|---|---|
| Executable filenames | `breeze-agent.exe` → `bl4ck-agent.exe` (also `-watchdog`, `-backup`, `-user-helper`, `-desktop-helper`) |
| Source command dirs | `agent/cmd/breeze-*` → `agent/cmd/bl4ck-*` (git mv) |
| Windows services | `BreezeAgent` → `Bl4ckAgent`, `BreezeWatchdog` → `Bl4ckWatchdog` (SCM name + display "BL4CK …") |
| Safe-Mode-Network key | `…\SafeBoot\Network\BreezeAgent` → `…\Bl4ckAgent` (matches service name) |
| Install folder | `C:\Program Files\Breeze` → `…\BL4CK`; `C:\ProgramData\Breeze` → `…\BL4CK` (Go `config_dir_windows.go` + WiX) |
| Scheduled task | `\Breeze\AgentUserHelper` → `\BL4CK\AgentUserHelper` |
| Event Log source | `BreezeAgent` → `Bl4ckAgent` |
| Registry marker | `HKLM\SOFTWARE\Breeze` → `…\BL4CK` |
| MSI identity | ProductName/Manufacturer/Feature/messages → "BL4CK …"; installer def `breeze.wxs` → `bl4ck.wxs`; output `dist/bl4ck-agent.msi` |
| File Properties metadata | go-winres `--product-name "BL4CK RMM Agent"`, `--original-filename bl4ck-*.exe` (Makefile + release.yml) |
| macOS identity | launchd/pkg `com.breeze.*` → `com.bl4ck.*`; Swift app `BreezeInstaller` → `Bl4ckInstaller` |
| Linux packaging | systemd units `breeze-agent*.service` → `bl4ck-agent*`, `/etc/breeze`→`/etc/bl4ck`, `/var/lib`,`/var/log`,`/run` |
| Web download + release | `breeze-agent-windows-amd64.exe` → `bl4ck-agent-…` in `installCommands.ts`, `downloadFilename.ts`, `RecoveryBootstrapTab.tsx`, `ServicesManager.tsx`, `uninstall.sh`, and `.github/workflows/release.yml` artifact names |

### How it works / why nothing breaks
- **Token-scoped rename, not blanket.** Replacements were specific tokens
  (`breeze-agent`, `BreezeAgent`, `com.breeze`, path stems) plus a case-sensitive
  `Breeze`→`BL4CK`. This deliberately did **not** touch:
  - The Go module path `github.com/breeze-rmm/agent` (compile-time only, never
    shown to a user — renaming it = 283-file churn for zero visible gain).
  - `BREEZE_*` environment-variable names (agent↔runtime contract; `BREEZE_PARAM_*`
    is read by customer scripts — renaming would break them). Case-sensitive rule
    left all-caps `BREEZE` untouched.
  - The Postgres role / RLS internals (different tree entirely).
- **Bootstrap enrollment still works.** The installer-filename token parser
  (`installer_filename.go`) matches only the `(TOKEN@HOST)` group anywhere in the
  name — it is prefix-agnostic, so "BL4CK Agent (TOKEN@HOST).msi" parses exactly
  like the old name.
- **Verified compiling:** `go build ./...` passes for **windows, linux, and
  darwin** after the rename.
- **UpgradeCode GUID kept stable** (no upgrade-continuity requirement, but keeping
  it is strictly safest).
- **Stray artifact noted:** `agent/breeze-backup` is a 61 MB macOS build binary
  accidentally committed long ago — not referenced by any build. Left as-is;
  recommend `git rm --cached agent/breeze-backup` separately.

### How to build + verify Phase 2 (you run these)
```bash
cd agent
# 1. (optional but needed for file-Properties branding) regenerate Windows resources
make build-winres                       # bakes "BL4CK RMM Agent" into each .exe
# 2. build the 5 windows binaries (into agent/ root, names build-msi.ps1 expects)
GOOS=windows GOARCH=amd64 go build -ldflags "-s -w" -o bl4ck-agent-windows-amd64.exe        ./cmd/bl4ck-agent
GOOS=windows GOARCH=amd64 go build -ldflags "-s -w" -o bl4ck-backup-windows-amd64.exe       ./cmd/bl4ck-backup
GOOS=windows GOARCH=amd64 go build -ldflags "-s -w" -o bl4ck-watchdog-windows-amd64.exe     ./cmd/bl4ck-watchdog
GOOS=windows GOARCH=amd64 go build -ldflags "-s -w -H windowsgui" -o bl4ck-user-helper-windows-amd64.exe ./cmd/bl4ck-user-helper
# 3. build the MSI
pwsh -File installer/build-msi.ps1 -Version 1.0.0
```
Then copy `dist/bl4ck-agent.msi` to `C:\Users\Public\`, rename to
`BL4CK Agent (TOKEN@HOST).msi` for bootstrap enrollment, install, and confirm:
Task Manager shows `bl4ck-agent.exe`, Services shows "BL4CK Agent", install path
is `C:\Program Files\BL4CK`, Add/Remove shows "BL4CK Agent — BL4CK RMM".

## Phase 2 FIX — wire-protocol headers must NOT be rebranded

### The bug
First install attempt failed with MSI error **1603** and rolled the whole install
back (no services, no processes, nothing in Add/Remove). The verbose MSI log
showed everything installing correctly under BL4CK, then:

```
ActionStart(Name=BootstrapEnroll)
WixSilentExec64: Error 0x80070001: Command line returned an error.
CustomAction BootstrapEnroll returned actual error code 1603
```

`BootstrapEnroll` is `Return="check"`, so a non-zero exit rolls back everything.
Reproducing the command without `--quiet` gave the real error:

```
bootstrap redeem failed: 400 {"error":"missing token"}
```

**Root cause:** the global `Breeze`→`BL4CK` rename renamed two **HTTP header
names**, which are a wire-protocol contract with the server. The server was NOT
rebranded (still v0.94.0), so it looked for the old header, found nothing, and
returned "missing token".

| File | Was | Reverted to |
|---|---|---|
| `internal/agentapp/bootstrap.go` | `X-BL4CK-Bootstrap-Token` | `X-Breeze-Bootstrap-Token` |
| `internal/watchdog/failover.go` | `X-BL4CK-Role` | `X-Breeze-Role` |

(+ matching test files.) Both now carry a comment warning future branding sweeps
not to rename them.

### How it was verified
Probed the live server with a deliberately bogus token so the real token stayed
unconsumed:

| Header sent | Response | Meaning |
|---|---|---|
| `X-BL4CK-Bootstrap-Token` | 400 | server saw **no** token |
| `X-Breeze-Bootstrap-Token` | 404 | token **received**, just not a real one |

400 → 404 proves the server now receives the token. `go test` passes for both
packages; all three platforms still build.

### The rule this establishes
**Anything that crosses the wire to the server must keep its original name.**
The full outbound header surface was enumerated to confirm nothing else broke:
`Accept`, `Authorization`, `Content-Encoding`, `Content-Type`, `User-Agent`,
`x-agent-reenrollment-token`, `X-API-KEY` — none contained "Breeze", so only the
two above were affected. API paths (`/api/v1/...`) were never touched.

Safe to rebrand (agent-internal, self-consistent): display strings, service
names, install paths, registry values the agent writes and reads itself
(`BL4CKManaged`, `BL4CKCreatedAUKey`), ETW session names, window class names, and
the outbound monitoring `User-Agent` (sent to customer-monitored endpoints, not
to the BL4CK API).

## Phase 3 — API rebrand + release-artifact alignment

### The latent break this fixed
Renaming the agent also renamed the **GitHub release asset names** in
`release.yml` (`breeze-agent.msi` → `bl4ck-agent.msi`, etc.), but `apps/api` was
outside that sweep and still asked for the old names. Artifact lookup is
**exact-match against the signed manifest — a mismatch is a hard failure, not a
fallback**. On the next release + deploy this would have broken:

- the dashboard's **"Download Installer"** button (503, API can't find the asset)
- **agent self-update** (`binarySync.ts` filename regex was `^breeze-…`)

Dormant until now only because the deployed server and the existing releases were
both still "breeze" — consistent with each other.

### What changed
| Area | Change |
|---|---|
| `services/binarySource.ts` | agent/helper artifact names → `bl4ck-*`; repo base → `github.com/bl4ckxr0sexd/bl4ck`; installer app → `BL4CK.Installer.app.zip` |
| `services/installerBuilder.ts` | `assetName: 'bl4ck-agent.msi'`, local paths, names inside the generated zip/.bat, `BL4CK Installer.app.zip` |
| `services/binarySync.ts` | self-update component names + filename regex `^bl4ck-${component}-…` |
| `routes/agents/download.ts` | generated install/uninstall scripts: `/usr/local/bin/bl4ck-agent`, `bl4ck-agent.service`, `SyslogIdentifier`, `/etc/bl4ck`, `/Library/Application Support/BL4CK` |
| `services/recoveryMediaService.ts`, `recoveryBootstrap.ts`, `routes/backup/bmr.ts` | recovery bundle + BMR helper binary names → `bl4ck-backup` |
| `apps/web` (AddDeviceModal, EnrollDeviceStep) | GitHub release base → `github.com/bl4ckxr0sexd/bl4ck` |

### Deliberately NOT renamed
- **`x-breeze-*` request headers** — wire protocol. Includes `x-breeze-csrf`,
  `x-breeze-signature`/`-timestamp`/`-event-id` (automation webhooks — renaming
  breaks every customer's configured sender), `x-breeze-bootstrap-token` (the
  server side of the header fixed in Phase 2), `x-breeze-mobile-device-id`.
- **`breeze-viewer-*`** — release still publishes the viewer under these names.
- **`breeze-billing`** — separate service.
- **Persisted/contract values** — `breeze-defaults` config sentinel, `_breeze-verify`
  DNS TXT prefix, S3 bucket names, `breeze-logs` index prefix, MCP tool names
  (`breeze-rmm`, `breeze-fleet-triage`, …). Renaming these invalidates existing
  records or breaks connected clients.

### Verification
Cross-checked all 13 artifact names the API fetches against what `release.yml`
publishes — **all match**, including the space-vs-dot distinction for the macOS
installer app (`BL4CK Installer.app.zip` as the manifest `assetName`, but
`BL4CK.Installer.app.zip` in the download URL, because GitHub rewrites spaces).

> Not run: `apps/api` unit tests — `node_modules` is not installed in this
> workspace. Changes are string-literal only and test fixtures were updated in
> the same sweep, but run `pnpm install && pnpm test --filter=@breeze/api`
> before deploying.

## Phase 4 — Helper app IPC alignment (another cross-component contract)

### The break
The agent rename changed the IPC endpoint names, but the Tauri Helper
(`apps/helper`) was outside the sweep, so the two no longer agreed — the Helper
could not reach the agent on **any** platform:

| Contract | Agent (renamed) | Helper (was) |
|---|---|---|
| Windows named pipe | `\\.\pipe\bl4ck-agent-ipc` | `\\.\pipe\breeze-agent-ipc` |
| macOS socket | `/Library/Application Support/BL4CK/agent.sock` | `…/Breeze/agent.sock` |
| Linux socket | `/var/run/bl4ck/agent.sock` | `/var/run/breeze/agent.sock` |
| Windows config | `C:\ProgramData\BL4CK\agent.yaml` | `…\Breeze\agent.yaml` |
| macOS/Linux config | `…/BL4CK/…`, `/etc/bl4ck/…` | `…/Breeze/…`, `/etc/breeze/…` |

Same class as the Phase-2 header bug — a contract between two components where
one side was renamed. **Difference:** this contract is between two pieces of our
OWN code (both rebuilt), not with the unchanged server, so the fix is to align
the Helper TO the agent's new names (not revert).

### What changed (`apps/helper`)
IPC pipe/socket + config paths → `bl4ck`/`BL4CK`; visible "Breeze Helper" strings
→ "BL4CK Helper" (`App.tsx`, `tauri.conf.json` productName, capabilities).

### Deliberately kept
- **`com.breezermm.helper`** Tauri bundle identifier — an OS identity contract
  (macOS TCC, updater); changing it resets permissions. Pre-existing `breezermm`
  namespace, not a regression from the rename. Visible name is rebranded; the id
  is not.
- **`2breeze.app`** portal-host check (`host_is_breeze_portal`) — must match the
  real deployment domain.
- `breeze-helper-username` localStorage key (persisted data) and test fixtures.

### Not verified here
`cargo` is not installed in this workspace, so the Rust crate was not
compile-checked. All changes are string-literal path swaps (no identifiers or
types touched), so compilation is unaffected — but build the Helper before
shipping it.
