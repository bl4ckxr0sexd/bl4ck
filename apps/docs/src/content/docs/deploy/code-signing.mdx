All BL4CK release binaries — the agent, the native viewer, and the Helper tray app — are code-signed to establish authenticity and prevent tampering warnings from operating system gatekeepers and endpoint security tools.

---

## Why Code Signing Matters

Unsigned binaries trigger security warnings on both Windows and macOS that block installation or execution. For an RMM agent that needs to run silently and with elevated privileges, code signing is essential:

| Platform | Without Signing | With Signing |
|----------|-----------------|--------------|
| **Windows** | SmartScreen blocks execution; AV products flag as suspicious; Group Policy may reject unsigned MSIs | Trusted publisher; silent installation works; SmartScreen passes |
| **macOS** | Gatekeeper quarantines the binary; users must manually override in System Settings; notarization fails | Gatekeeper passes; notarization succeeds; MDM deployment works |

## Signed Artifacts

The CI/CD release pipeline signs the following artifacts:

| Artifact | Platform | Format | Signing Method |
|----------|----------|--------|----------------|
| BL4CK Agent | Windows | `.exe`, `.msi` | Azure Code Signing (EV certificate) |
| BL4CK Agent | macOS | `.pkg`, binary | Apple Developer ID + notarization |
| BL4CK Viewer | Windows | `.exe`, `.msi` | Azure Code Signing (EV certificate) |
| BL4CK Viewer | macOS | `.app`, `.dmg` | Apple Developer ID + notarization |
| BL4CK Helper | Windows | `.exe` | Azure Code Signing (EV certificate) |
| BL4CK Helper | macOS | `.app` | Apple Developer ID + notarization |

## Windows Code Signing

Windows binaries are signed using **Azure Code Signing** with an Extended Validation (EV) certificate. The signing happens in the GitHub Actions release workflow.

### How it works

<Steps>

1. The release workflow builds the Windows binaries (agent `.exe`, MSI installer, viewer, helper).

2. The `AzureSignTool` utility authenticates to Azure Key Vault using a service principal.

3. Each binary is signed with the EV certificate stored in Azure Key Vault.

4. The MSI installer is signed separately after the inner `.exe` is signed.

5. Signed artifacts are uploaded to the GitHub release.

</Steps>

### WiX v4 MSI Packaging

The Windows agent MSI is built using WiX v4. The MSI bundles the signed agent binary, configures the Windows service, and sets appropriate file permissions. The MSI itself is then signed as a separate step.

<Aside>
  Azure Code Signing uses HSM-backed keys — the private key never leaves Azure Key Vault. This meets the CA/Browser Forum requirements for EV code signing certificates.
</Aside>

## macOS Code Signing & Notarization

macOS binaries are signed with an **Apple Developer ID Application** certificate and then submitted to Apple's notarization service.

### How it works

<Steps>

1. The release workflow builds the macOS universal binaries (arm64 + amd64).

2. Each binary is signed with `codesign` using the Developer ID certificate from the CI keychain.

3. The signed binary is packaged into a `.pkg` or `.app` bundle.

4. The package is submitted to Apple's notarization service via `notarytool`.

5. The workflow waits for notarization to complete (timeout: 30 minutes).

6. Once notarized, the package is stapled with `stapler` so it can be verified offline.

</Steps>

<Aside type="caution">
  Notarization can take several minutes. The release workflow uses a 30-minute timeout to accommodate Apple's processing queue. If notarization fails with `UnknownError`, the workflow retries — this status code sometimes appears transiently with newly provisioned certificates.
</Aside>

### Gatekeeper Verification

End users can verify that a downloaded binary passes Gatekeeper:

```bash
spctl --assess --verbose /path/to/breeze-agent
# Expected: accepted, source=Notarized Developer ID
```

## Installer Download Signing

When you download a pre-configured MSI installer from the dashboard or a public installer link, BL4CK can automatically code-sign the installer before delivery. This ensures that the patched MSI — which contains your enrollment key and site assignment — is trusted by Windows SmartScreen and antivirus tools.

### How it works

The API server sends the patched MSI to a dedicated Windows signing service (running `signtool` with Azure Trusted Signing) over HTTPS. The signing service returns the signed MSI, which is then delivered to the browser. The entire process adds a few seconds to the download.

### Configuration

Set three environment variables to enable installer signing:

| Variable | Description |
|----------|-------------|
| `MSI_SIGNING_URL` | URL of the signing service endpoint |
| `MSI_SIGNING_CF_ACCESS_ID` | Cloudflare Access service token Client ID (optional — only needed if the signing service is behind Cloudflare Access) |
| `MSI_SIGNING_CF_ACCESS_SECRET` | Cloudflare Access service token Client Secret |

When these variables are unset, installer downloads are delivered without an additional signature. The MSI template itself is still signed during the CI build, so the base installer is always signed — this step re-signs the MSI after enrollment data is embedded.

See [Environment Variables](/deploy/environment/) for the full list.

## Verifying Signatures

### Windows

```powershell
# Check digital signature on a binary
Get-AuthenticodeSignature "C:\Program Files\Breeze\breeze-agent.exe"

# Expected: Status = Valid, SignerCertificate shows LanternOps
```

### macOS

```bash
# Verify code signature
codesign --verify --deep --strict /usr/local/bin/breeze-agent

# Check notarization
spctl --assess --verbose /usr/local/bin/breeze-agent
```

## Troubleshooting

**Windows SmartScreen still warns after signing.**
SmartScreen reputation is built over time. Newly signed binaries from a new publisher may still show a warning until enough users have downloaded and run the binary. EV certificates build reputation faster than standard OV certificates. If the warning persists, verify the signature with `Get-AuthenticodeSignature` to confirm the binary is properly signed.

**macOS notarization timeout.**
Apple's notarization service can be slow during peak times. The workflow allows up to 30 minutes. If it times out, re-run the release workflow — the binary does not need to be rebuilt, only re-submitted.

**"Developer cannot be verified" on macOS.**
The binary was not notarized, or the stapled ticket is missing. Re-download the binary from the official release page. If installing via MDM, ensure the MDM profile allows the Developer ID. Users can temporarily override Gatekeeper via **System Settings → Privacy & Security** but this should not be necessary for properly signed releases.

**Antivirus flags the agent after signing.**
Some endpoint security products flag new binaries regardless of signature status. See [Antivirus Exceptions](/deploy/antivirus-exceptions/) for recommended exclusions per platform.
